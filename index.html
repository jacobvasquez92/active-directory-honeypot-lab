// Phase 1

Isolating CLIENT1 as a Honeypot

Network segmentation ensures the exposed VM cannot pivot to your host or domain controller, while remaining reachable from the internet.

STEP 1 Create VMnet3 in VMware Virtual Network Editor
  1. Open VMware Workstation → Edit → Virtual Network Editor
  2. Click Add Network → select VMnet3
  3. Set type to Host-only
  4. Disable DHCP on VMnet3 — IPs on this segment will be static
  5. Uncheck "Connect a host virtual adapter to this network" for VMnet3
  6. Click Apply → OK
ℹ
VMnet1 and VMnet3 are isolated virtual switches by default — they have no routing path between them unless a VM has NICs on both networks (which CLIENT1 will). The steps below explicitly block that cross-network path.
STEP 2 Add NICs to CLIENT1 — Three-Adapter Layout

Shut down CLIENT1, then go to VM Settings → Add → Network Adapter twice to create the full layout:

NICVMware NetworkIP AssignmentPurpose
NIC 1VMnet1 (Host-only)DHCP from DCDomain membership, AD authentication
NIC 2Bridged (auto-detect)DHCP reservation on routerInternet-facing honeypot surface
NIC 3VMnet3 (Host-only)Static: 192.168.100.10Wazuh log forwarding only
⚡
NIC 2 Internet Exposure: The Bridged adapter assigns CLIENT1 its own LAN IP from your router (e.g. 192.168.1.150). Log into your router admin page and create a DHCP reservation binding CLIENT1's MAC address to a fixed IP. Then add port-forwarding rules for ports 3389, 445, 21, and 23 pointing to that IP. ⚠ See TS-01
STEP 3 Enforce Network Isolation — Block Cross-VMnet Routing

Because CLIENT1 has NICs on both VMnet1 and VMnet3, you must prevent it from routing traffic between them. Run these commands on CLIENT1 in PowerShell as Administrator.

Part A — Remove gateway from VMnet3 adapter

# Identify adapter names first
Get-NetAdapter

# Set static IP on VMnet3 with NO gateway
netsh interface ip set address name="Ethernet 3" static 192.168.100.10 255.255.255.0
# Note: omitting the gateway parameter means none is set

Part B — Disable IP forwarding on CLIENT1

Set-NetIPInterface -Forwarding Disabled -AddressFamily IPv4

Part C — Firewall rules blocking cross-network traffic

# Block VMnet3 adapter from reaching VMnet1 subnet
New-NetFirewallRule -DisplayName "Block VMnet3 to VMnet1" `
  -Direction Outbound `
  -RemoteAddress 192.168.1.0/24 `
  -InterfaceAlias "Ethernet 3" `
  -Action Block -Profile Any

# Block VMnet1 adapter from reaching VMnet3 subnet
New-NetFirewallRule -DisplayName "Block VMnet1 to VMnet3" `
  -Direction Outbound `
  -RemoteAddress 192.168.100.0/24 `
  -InterfaceAlias "Ethernet 1" `
  -Action Block -Profile Any

Part D — Protect DC from VMnet3 (run on DC)

New-NetFirewallRule -DisplayName "Block Honeypot VMnet3" `
  -Direction Inbound `
  -RemoteAddress 192.168.100.0/24 `
  -Action Block -Profile Any

Part E — Verify isolation

# This should FAIL (DC unreachable from VMnet3 interface)
Test-NetConnection -ComputerName 192.168.1.1 -Source 192.168.100.10

# This should SUCCEED (Wazuh Manager reachable)
Test-NetConnection -ComputerName 172.16.0.101 -Port 1514

// Phase 2

Installing Snort IDS on CLIENT1

Snort monitors live network traffic on CLIENT1's internet-facing adapter and writes alerts to a log file that Wazuh will later ingest.

STEP 1 Install Npcap (Packet Capture Driver)
  1. Download Npcap from https://npcap.com/#download
  2. Run installer — check these options:
    • ✅ Install Npcap in WinPcap API-compatible mode (required for Snort)
    • ✅ Restrict Npcap driver's access to Administrators only
  3. Reboot CLIENT1 after install
STEP 2 Install Snort 2.9.x
  1. Download from https://www.snort.org/downloads — choose Snort 2.9.x Windows binary (not Snort 3)
  2. Run installer, accept defaults, install to C:\Snort

Expected directory structure after install:

C:\Snort\
  bin\        ← snort.exe
  etc\        ← snort.conf
  log\        ← alerts written here
  rules\      ← rule files
  lib\        ← dynamic preprocessors
STEP 3 Download Community Rules & Create Support Files
  1. Download Snort Community Rules from https://www.snort.org/downloads#rules (no account needed)
  2. Extract and copy community.rules to C:\Snort\rules\community.rules
REM Create required empty files
type nul > C:\Snort\rules\local.rules
type nul > C:\Snort\etc\white_list.rules
type nul > C:\Snort\etc\black_list.rules
STEP 4 Configure snort.conf

Open C:\Snort\etc\snort.conf in Notepad as Administrator. Make these edits:

Network variables (~line 45):

ipvar HOME_NET 192.168.1.0/24
ipvar EXTERNAL_NET !$HOME_NET

Rule paths (~line 104):

var RULE_PATH C:\Snort\rules
var SO_RULE_PATH C:\Snort\rules
var PREPROC_RULE_PATH C:\Snort\rules
var WHITE_LIST_PATH C:\Snort\etc
var BLACK_LIST_PATH C:\Snort\etc
⚡
If WHITE_LIST_PATH / BLACK_LIST_PATH appear twice in the file, delete the second occurrence (which sets ../rules) — it overrides your correct path and causes concatenation errors. ⚠ See TS-04

Fix dynamic preprocessor paths ⚠ See TS-02 — find and replace all Linux-style paths:

Find (Linux path)Replace with (Windows path)
/usr/local/lib/snort_dynamicpreprocessor/C:\Snort\lib\snort_dynamicpreprocessor
/usr/local/lib/snort_dynamicengine/C:\Snort\lib\snort_dynamicengine
/usr/local/lib/snort_dynamicrules/C:\Snort\lib\snort_dynamicrules
 
REM Create the folders if they don't exist
mkdir C:\Snort\lib\snort_dynamicpreprocessor
mkdir C:\Snort\lib\snort_dynamicengine
mkdir C:\Snort\lib\snort_dynamicrules

Comment out all rule includes except yours — use Notepad Ctrl+H:

  • Find: include $RULE_PATH/ → Replace: # include $RULE_PATH/ (comments all out)
  • Then remove the # from only community.rules and local.rules
  • Comment out any bare text lines like dynamic library rules → add # in front ⚠ See TS-05

Final rules section should look like:

include $RULE_PATH\community.rules
include $RULE_PATH\local.rules

# decoder and preprocessor event rules
# include $PREPROC_RULE_PATH/preprocessor.rules
# include $PREPROC_RULE_PATH/decoder.rules

# dynamic library rules
# include $SO_RULE_PATH/bad-traffic.rules
# (...all SO rules commented out...)

include threshold.conf
STEP 5 Add Honeypot Detection Rules to local.rules

Open C:\Snort\rules\local.rules and add these rules. Ensure all quotes are straight quotes — not curly/smart quotes from copy-paste.

# RDP brute force detection
alert tcp any any -> $HOME_NET 3389 (msg:"RDP Brute Force Attempt"; threshold:type threshold, track by_src, count 5, seconds 60; sid:1000001; rev:1;)

# SMB scan detection
alert tcp any any -> $HOME_NET 445 (msg:"SMB Scan Detected"; flags:S; threshold:type threshold, track by_src, count 10, seconds 10; sid:1000002; rev:1;)

# ICMP ping (no threshold — fires on every ping)
alert icmp any any -> $HOME_NET any (msg:"ICMP Ping Test"; sid:1000003; rev:1;)

# Telnet connection
alert tcp any any -> $HOME_NET 23 (msg:"Telnet Connection Attempt"; sid:1000004; rev:1;)

# FTP connection
alert tcp any any -> $HOME_NET 21 (msg:"FTP Connection Attempt"; sid:1000005; rev:1;)
ℹ
The original ICMP rule used threshold:type threshold, count 5, seconds 5 which requires 5 pings to trigger. Removing the threshold means every ping fires an alert — better for initial testing. ⚠ See TS-07
STEP 6 Find Interface Index & Test Config 
REM List all interfaces — note the index of your Bridged adapter (NIC 2)
C:\Snort\bin\snort.exe -W

Then test the config (no traffic captured, just validates):

C:\Snort\bin\snort.exe -T -i 2 -c C:\Snort\etc\snort.conf

Expected output: Snort successfully validated the configuration!

STEP 7 Run Snort in IDS Mode 
C:\Snort\bin\snort.exe -i 2 -c C:\Snort\etc\snort.conf -l C:\Snort\log -A full -K ascii
FlagMeaning
-i 2Listen on interface index 2 (bridged adapter)
-cPath to snort.conf
-lWrite logs to C:\Snort\log\
-A fullFull alert output with packet headers
-K asciiHuman-readable log format

Test by pinging CLIENT1's bridged IP from another machine, then check:

type C:\Snort\log\alert.ids

Optional — Run as Windows service ⚠ See TS-08

REM If service install fails, use this minimized background approach instead:
start /min "Snort IDS" "C:\Snort\bin\snort.exe" -i 2 -c "C:\Snort\etc\snort.conf" -l "C:\Snort\log" -A full -K ascii

// Phase 3

Deploying Wazuh SIEM

Wazuh provides centralized log management, alerting, and a dashboard. It runs on a separate Ubuntu VM and receives logs from both CLIENT1 and the DC.

STEP 1 Deploy Wazuh Manager VM (Ubuntu 22.04 LTS)
  1. Create a new VM: Ubuntu 22.04 LTS · 4GB RAM · 2 vCPUs · 50GB disk
  2. Assign NIC to VMnet1 (same as DC and CLIENT1 NIC1)
  3. After OS install, run the Wazuh all-in-one installer:
curl -sO https://packages.wazuh.com/4.7/wazuh-install.sh
sudo bash wazuh-install.sh -a
⚡
If you get an OS compatibility error, add the -i flag to bypass the check. This is safe for lab use. ⚠ See TS-09

After install completes, retrieve your dashboard credentials:

sudo tar -O -xvf wazuh-install-files.tar wazuh-install-files/wazuh-passwords.txt

Access the dashboard from your host browser: https://172.16.0.101 (accept the self-signed cert warning). Login with username: admin and the generated password. ⚠ See TS-10

STEP 2 Install Wazuh Agent on CLIENT1

On CLIENT1 in PowerShell as Administrator:

# Step 1 — Download the agent MSI (must match your manager version: 4.7.5)
$client = New-Object System.Net.WebClient
$client.DownloadFile(
  "https://packages.wazuh.com/4.x/windows/wazuh-agent-4.7.5-1.msi",
  "C:\Users\admin\Downloads\wazuh-agent.msi"
)

# Step 2 — Verify file size (~50MB)
Get-Item "C:\Users\admin\Downloads\wazuh-agent.msi" | Select-Object Name, Length
⚡
If the install fails with "this installation package could not be opened", the MSI downloaded incompletely. Check file size — it should be ~50MB. Delete and re-download. ⚠ See TS-11
# Step 3 — Install with verbose logging (shows errors instead of failing silently)
msiexec.exe /i "C:\Users\admin\Downloads\wazuh-agent.msi" `
  WAZUH_MANAGER="172.16.0.101" `
  WAZUH_AGENT_NAME="CLIENT1-Honeypot" `
  /l*v "C:\wazuh-install.log"

# Step 4 — Start the service
NET START WazuhSvc

# Step 5 — Verify connection
Get-Service WazuhSvc
Get-Content "C:\Program Files (x86)\ossec-agent\ossec.log" -Tail 20

Look for Connected to the server in the log output. ⚠ See TS-12

Test connectivity to manager on port 1514:

Test-NetConnection -ComputerName 172.16.0.101 -Port 1514
# Expected: TcpTestSucceeded: True
STEP 3 Install Wazuh Agent on DC

Repeat the same process on DC — gives full domain visibility for lateral movement detection:

msiexec.exe /i "C:\Users\admin\Downloads\wazuh-agent.msi" `
  WAZUH_MANAGER="172.16.0.101" `
  WAZUH_AGENT_NAME="DC-DomainController" `
  /l*v "C:\wazuh-install-dc.log"
NET START WazuhSvc
STEP 4 Configure Wazuh to Ingest Snort Alerts

On CLIENT1, open C:\Program Files (x86)\ossec-agent\ossec.conf in Notepad as Administrator. ⚠ See TS-13

  1. Press Ctrl+F and search for </ossec_config>
  2. Insert this block immediately before the closing tag:
  <localfile>
    <log_format>snort-full</log_format>
    <location>C:\Snort\log\alert.ids</location>
  </localfile>

</ossec_config>
  1. File → Save
  2. Restart the agent:
Restart-Service WazuhSvc
Get-Content "C:\Program Files (x86)\ossec-agent\ossec.log" -Tail 30
# Look for: snort or alert.ids in the log output
STEP 5 Install Sysmon for Deep Endpoint Visibility

Sysmon captures process creation, network connections, file hashes, and registry changes — exactly what attackers leave behind.

# Download Sysmon (Microsoft Sysinternals)
Invoke-WebRequest -Uri "https://download.sysinternals.com/files/Sysmon.zip" -OutFile sysmon.zip
Expand-Archive sysmon.zip -DestinationPath C:\Sysmon

# Download SwiftOnSecurity config (industry-standard baseline)
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/SwiftOnSecurity/sysmon-config/master/sysmonconfig-export.xml" `
  -OutFile C:\Sysmon\sysmon-config.xml

# Install
C:\Sysmon\Sysmon64.exe -accepteula -i C:\Sysmon\sysmon-config.xml

Wazuh automatically reads Sysmon events from the Windows Event Log via its built-in rules.

Attack signatures this lab captures:

  • 🔴 RDP/SMB brute-force — Snort threshold alerts + Windows Event ID 4625 (failed logon)

  • 🟡 Lateral movement — Sysmon + Wazuh unusual process spawning
  • 🟡 Persistence — Registry run key changes, scheduled task creation
